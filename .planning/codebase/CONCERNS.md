# Codebase Concerns

**Analysis Date:** 2026-05-04

## Tech Debt

**ML Models Missing from Repository:**
- Issue: The `models/` directory exists but is empty (0 bytes total). All `.pkl`, `.json`, and `.onnx` model files are gitignored via `models/*.pkl`, `models/*.json`, `models/*.onnx`, `models/*.keras`. The API handler (`frontend/api/predict.py`) hard-fails if model files are not present at runtime.
- Files: `/Users/nurbek/HardWork/fari_diploma/models/`, `frontend/api/predict.py` lines 42-67
- Impact: The real-time `/api/predict` endpoint is non-functional in any fresh clone or deployment without running `scripts/train_models.py` first. No README or CI step automates this.
- Fix approach: Either commit small quantized model files (check size limits), add a model download script, or document the exact training-to-deploy workflow in a setup script.

**Static JSON Data is Pre-generated and Hardcoded:**
- Issue: All dashboard data (`frontend/public/data/**/*.json`) is generated once by `scripts/generate_dashboard_data.py` and committed as static files. The `training_history.json` files are literal placeholders: `{"note": "placeholder"}` and `{"note": "Run train_models.py to generate autoencoder loss history"}`. The data does not reflect live model state.
- Files: `frontend/public/data/creditcard/training_history.json`, `frontend/public/data/paysim/training_history.json`, `scripts/generate_dashboard_data.py` lines 327-334
- Impact: Any re-training of models requires manually re-running `generate_dashboard_data.py` and re-committing the JSON files. Training history charts/UI would show empty/stub data.
- Fix approach: Run `generate_dashboard_data.py` as a post-train step, or store training history during `train_models.py` and read it back in `generate_dashboard_data.py`.

**Monkey-patched `np.histogram` Never Applied:**
- Issue: `scripts/generate_dashboard_data.py` defines a `json_histogram` function to monkey-patch `np.histogram` (lines 372-380) but never calls `np.histogram = json_histogram`. For the PaySim branch the explicit `_hist_to_json` helper is used correctly; for the Credit Card branch `np.histogram` is called directly (line 304-310) and the raw numpy tuple result is passed to `save_json`, which would fail serialization.
- Files: `scripts/generate_dashboard_data.py` lines 302-310, 372-380
- Impact: Running `generate_dashboard_data.py` for creditcard distributions would raise a `TypeError` during `json.dump` since numpy arrays are not JSON serializable. The committed `distributions.json` was likely generated by a different version of the script.
- Fix approach: Apply the monkey-patch before the Credit Card data generation call, or replace the direct `np.histogram` call at lines 304-310 with `_hist_to_json` (already defined at line 359).

**`filter` State Not Reset on Dataset Switch (Transactions Page):**
- Issue: When the user switches datasets via `DatasetSwitcher`, the `useEffect` in `TransactionsPage` resets `page` and `search` but does NOT reset `filter` (the `all`/`fraud`/`normal` filter button state).
- Files: `frontend/src/pages/Transactions.tsx` lines 20-25
- Impact: If a user switches from creditcard to paysim while `filter === "fraud"`, the fraud filter remains active against the new dataset's transactions, which is unexpected UX. Not a crash but a logical bug.
- Fix approach: Add `setFilter("all")` inside the `useEffect` that depends on `[dataset]`.

**Unused Dependency: `@tanstack/react-table`:**
- Issue: `@tanstack/react-table` is listed in `package.json` dependencies but zero imports of it appear anywhere in `frontend/src/`. All tables are implemented with raw `<table>` HTML.
- Files: `frontend/package.json`
- Impact: Adds ~180KB to the dependency tree unnecessarily, increases install time, and creates a maintenance surface for a library that is not used.
- Fix approach: Remove `@tanstack/react-table` from `package.json` and run `npm install`.

**`"use client"` Directive in Vite/React Project:**
- Issue: Several chart components (`frontend/src/components/charts/roc-chart.tsx`, `pr-chart.tsx`, `f1-bar-chart.tsx`, `class-balance-chart.tsx`) have `"use client"` as their first line. This directive is a Next.js App Router concept and is meaningless in a Vite+React project; it is silently ignored.
- Files: `frontend/src/components/charts/roc-chart.tsx` line 1, `frontend/src/components/charts/pr-chart.tsx` line 1, `frontend/src/components/charts/f1-bar-chart.tsx` line 1, `frontend/src/components/charts/class-balance-chart.tsx` line 1
- Impact: No runtime breakage, but signals the project was migrated from Next.js and these files were not cleaned up. Creates confusion for contributors.
- Fix approach: Remove all `"use client"` directives from chart and component files.

**`noUnusedLocals` and `noUnusedParameters` Disabled:**
- Issue: `frontend/tsconfig.json` explicitly sets `"noUnusedLocals": false` and `"noUnusedParameters": false`, disabling TypeScript's ability to flag dead code.
- Files: `frontend/tsconfig.json` lines 17-18
- Impact: Unused imports, variables, and function parameters accumulate silently. Already visible: `@tanstack/react-table` import exists in `package.json` while no code uses it.
- Fix approach: Set both to `true` and resolve any reported errors; this catches dead code at compile time.

## Security Considerations

**No Input Validation on `/api/predict` Endpoint:**
- Risk: The `dataset` field from the POST body is accepted without allowlist validation. Any value other than `"creditcard"` maps to the `"ps"` (PaySim) tag via a simple ternary, meaning an attacker sending `dataset: "anything"` loads PaySim models silently with no error. The `features` dict is passed directly to `predict()` with no type checking or size limits.
- Files: `frontend/api/predict.py` lines 148-152
- Current mitigation: None.
- Recommendations: Allowlist `dataset` values explicitly (raise 400 on invalid); validate that `features` is a dict of numeric values with a bounded key count; add a `Content-Length` upper bound guard.

**No Request Size Limit on API Handler:**
- Risk: `content_length = int(self.headers.get("Content-Length", 0))` reads whatever size is declared in the header. An attacker can send a very large `Content-Length` to cause excessive memory allocation in `rfile.read(content_length)`.
- Files: `frontend/api/predict.py` line 145-146
- Current mitigation: None in code; Vercel platform may impose limits at infrastructure level, but this is not relied upon explicitly.
- Recommendations: Add a guard such as `if content_length > 65536: return 400` before reading the body.

**No CORS Headers on API:**
- Risk: The Python `BaseHTTPRequestHandler`-based handler sets no `Access-Control-Allow-Origin` headers. In a cross-origin deployment scenario, browsers would block preflight requests. If the API is deployed alongside the Vite SPA on the same Vercel domain this is low risk, but it is fragile if hosting changes.
- Files: `frontend/api/predict.py` lines 154-157
- Current mitigation: Co-hosted on Vercel so same-origin currently applies.
- Recommendations: Add CORS headers to support potential future separation of frontend and API hosting.

**No Authentication on Prediction API:**
- Risk: The `/api/predict` endpoint has no authentication or rate limiting. Any client can call it indefinitely, incurring Vercel function invocation costs and potentially causing abuse.
- Files: `frontend/api/predict.py`
- Current mitigation: None.
- Recommendations: For a diploma demo this is acceptable, but production use would require API key validation or Vercel's built-in rate limiting.

## Performance Bottlenecks

**Cold Start Latency for ML Model Loading:**
- Problem: On Vercel serverless functions, each new function instance must load models from disk on first invocation. `_load_model()` calls `joblib.load()` for the scaler and Isolation Forest, `xgb_model.load_model()` for XGBoost, and `ort.InferenceSession()` for the ONNX autoencoder. Combined this could take several seconds on cold start.
- Files: `frontend/api/predict.py` lines 31-81
- Cause: Vercel Python functions do not persist state between invocations in most configurations; the `_models = {}` module-level cache only helps within a single warm instance lifetime.
- Improvement path: Minimize model file sizes (quantize XGBoost, export smaller ONNX); note warm instance reuse in Vercel's execution model is non-guaranteed.

**Transactions JSON Loaded Entirely into Memory:**
- Problem: `getTransactions()` in `frontend/src/lib/data.ts` fetches the entire `transactions.json` file (156KB creditcard, 222KB paysim) into memory on every page visit (first visit only due to cache). All filtering, pagination, and search run on the full in-memory array.
- Files: `frontend/src/lib/data.ts` lines 43-45, `frontend/src/pages/Transactions.tsx` lines 27-43
- Cause: No server-side filtering; static JSON approach requires full client-side data.
- Improvement path: For a diploma demo the sizes are acceptable. If data grows, consider pre-paginated JSON files or a backend filter endpoint.

**ROC/PR Curve Data Rendered with Full Point Arrays:**
- Problem: `roc-chart.tsx` constructs a new data array on every render by mapping over `data[model].fpr` per model. With multiple models and up to 200 points each this runs on every React render cycle.
- Files: `frontend/src/components/charts/roc-chart.tsx` lines 10-21
- Cause: No `useMemo` wrapping the data transformation.
- Improvement path: Wrap the merged data construction in `useMemo` keyed on `data`.

## Fragile Areas

**`generate_dashboard_data.py` Requires Raw CSV Files:**
- Files: `scripts/generate_dashboard_data.py` lines 50-76
- Why fragile: The script hard-`sys.exit()`s if `creditcard.csv` or the PaySim CSV are not found. Both CSVs are gitignored (100MB+ files). Regenerating dashboard data requires downloading the original Kaggle datasets and placing them at the repository root, which is not documented anywhere in the codebase.
- Safe modification: Only modify the committed static JSON files in `frontend/public/data/`; avoid running `generate_dashboard_data.py` without the source CSVs present.

**`DatasetContext` Has No Persistence:**
- Files: `frontend/src/hooks/use-dataset.tsx`
- Why fragile: The selected dataset resets to `"creditcard"` on every page refresh because state is held only in React memory with no `localStorage` persistence.
- Safe modification: Add `localStorage` read/write around `useState` and `setDataset` to persist the selection across reloads.

**API Predict Silently Falls Back to PaySim on Unknown Dataset:**
- Files: `frontend/api/predict.py` line 149: `dataset_tag = "cc" if dataset == "creditcard" else "ps"`
- Why fragile: Any typo or unknown dataset string silently loads PaySim models. No 400 error is returned; the user receives a prediction result that may be for the wrong dataset.
- Safe modification: Replace with an explicit allowlist check and return HTTP 400 for invalid inputs.

**`ScoreBadge` Assumes Score Is Always Defined:**
- Files: `frontend/src/pages/Transactions.tsx` lines 280-290
- Why fragile: `ScoreBadge` calls `score.toFixed(3)` without checking if `score` is `undefined`. If a transaction's `scores` object is missing a model key (e.g. `scores["Autoencoder"]` when the autoencoder was not loaded during data generation), this will throw `Cannot read properties of undefined`.
- Safe modification: Guard with `score != null ? score.toFixed(3) : "N/A"`.

- Problem: All `useEffect` hooks in Dashboard, Models, Analytics, and Transactions pages call `.then(setState)` with no `.catch()` handler. If any JSON file fails to load (network error, 404, malformed JSON), the component silently stays in its loading/empty state with no user-facing error message.
- Files: `frontend/src/pages/Dashboard.tsx` lines 20-22, `frontend/src/pages/Models.tsx` lines 17-19, `frontend/src/pages/Analytics.tsx` lines 24-27, `frontend/src/pages/Transactions.tsx` lines 21-22
- Blocks: Diagnosing production data load failures; providing user feedback.

**No Loading State for Most Pages:**
- Problem: Only `Dashboard.tsx` renders a `<LoadingSkeleton />` while data loads. Models, Analytics, Transactions, and Predict pages render empty tables/charts with no loading indicator while their `useEffect` fetches complete.
- Files: `frontend/src/pages/Models.tsx`, `frontend/src/pages/Analytics.tsx`, `frontend/src/pages/Transactions.tsx`
